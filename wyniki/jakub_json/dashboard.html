Browser Fingerprint Analysis

Metrics.
Trackability % — 100% = same fingerprint ID every session (permanently trackable). 0% = new ID every session.
Volatility — share of consecutive session pairs where a hash changed. 0 = never changes (strong identifier). 1 = changes every session (not usable for tracking).
Unique ratio — distinct hash values ÷ total sessions. 1/N = one value for all sessions (maximally stable). N/N = each session unique (maximally random).
Trust score — CreepJS score 0–100 measuring internal API consistency. 0 + many lies = JS proxy detected (CanvasBlocker-style). This is itself a fingerprinting signal.
Lie categories — JS classes where the same property returned different values when called via different code paths. Method count = how many methods in that class were caught.

2 — Hash Stability: Summary Signals

The ten hashes in summary are the primary fingerprinting signals that CreepJS exports. Each represents a different hardware or software surface. Canvas 2D / WebGL / Audio are rendering-based hashes — they reflect GPU and driver behaviour and are highly stable on unmodified browsers. Anti-fingerprinting tools target these specifically. Navigator / Worker encode browser capabilities and platform info. Fonts hashes the exact set of enumerable fonts — extremely stable and uniquely identifying. Timezone and Screen are lower-entropy but still contribute to fingerprints. WebRTC / Status are often null (blocked or unavailable). A signal with volatility 0 and unique ratio 1/N is a permanent tracking vector. One with volatility 1 and ratio N/N is effectively noise — different every session, unusable for linking.

3 — Hash Stability: Internal Sub-object Hashes

CreepJS hashes each internal data object in fullFingerprint separately and stores a $hash field on each. These 24 hashes are more granular than the summary hashes. Client Rects hashes sub-pixel layout rendering — GPU-dependent, very stable, a strong cross-session identifier even if canvas is spoofed. Maths hashes floating-point arithmetic results — CPU/FPU-specific, stable across reboots. Resistance hashes which extension APIs are intercepted — this directly fingerprints the specific anti-fingerprinting extension and its configuration. Two users of the same extension may share this hash, making it a crowd-anonymity signal rather than a unique identifier. Features and CSS reflect supported browser APIs and rendering engine version. Lies / Trash / Captured Errors encode the spoofing footprint itself.

5 — Trust Score & Lies Over Time

Trust score (left) measures API self-consistency. A normal unmodified browser scores 85–100. A score of 0 with ~280 lies is the signature of CanvasBlocker: it intercepts Canvas / WebGL / Audio APIs via JavaScript Proxy objects, and CreepJS detects this by querying the same property through multiple independent code paths and comparing results. Critically, this makes the browser uniquely identifiable as "running CanvasBlocker" — it trades canvas-hash tracking for proxy-detection tracking. Tor Browser scores 60–75 because it modifies outputs at the browser engine level (no detectable proxies), which is harder to detect but still slightly inconsistent with a stock browser. Lies (right) — each lie is one API method that returned inconsistent values. A spike to 200+ lies in a clean-Firefox session likely indicates the CanvasBlocker extension was briefly active or a different profile was accidentally used.

6 — Lie Categories: Which APIs Are Being Intercepted?

When a spoofing tool intercepts a JS API via Proxy, the native Function.prototype.toString() returns a different string for the proxied function than for the native one — this is why Function always appears at the top when CanvasBlocker is active. The other categories correspond to specific spoofing targets: CanvasRenderingContext2D and HTMLCanvasElement for canvas spoofing; AnalyserNode / AudioBuffer for audio spoofing; Math and Date for entropy reduction. Each number is the average count of intercepted methods in that class per session. Profiles with no proxy-based spoofing will have no rows here.

9 — Reference: What Every Metric & Hash Means

Plain-language explanation of every number, hash, and signal in this report.

Top-level metrics

MetricWhat it measuresHow to read it
Trackability % Derived from fingerprint ID (FP ID) stability. 100% = the same FP ID appeared in every session — a cross-session tracker can permanently re-identify this browser. 0% = a new FP ID every session — no two sessions are linkable by FP ID alone. 90–100% permanently trackable. 40–89% partially trackable. 0–39% hard to track.
Unique FP IDs / total Raw count of distinct fingerprint IDs across all sessions, over total sessions. The FP ID is CreepJS's top-level hash of the entire fingerprint object. 1/23 = always the same identity. 23/23 = new identity every session.
Trust score CreepJS score 0–100. Measures internal API self-consistency: the same property is queried through multiple independent code paths and the results are compared. An unmodified browser scores 85–100. Score of 0 with 200+ lies = JS proxy-based spoofing detected (CanvasBlocker). The score is itself a fingerprinting signal — a consistently low score is identifying. 85–100: normal browser, undetectable. 50–84: some modification. 0–49: heavy modification. Exactly 0 = proxy intercept confirmed.
Lies / session Number of API methods that returned inconsistent values when called through different code paths in one session. Each lie = one method caught returning a spoofed value. 200+ lies = CanvasBlocker or similar JS proxy tool. The lie count and pattern are themselves a stable fingerprinting signal. 0: no spoofing. 1–20: engine-level modification (Tor, resistFingerprinting). 200+: proxy-based spoofing.
Stealth rating CreepJS estimate of how detectable the browser's modifications are to a fraud-detection or bot-detection system. Based on patterns in API responses, timing anomalies, and other signals. A consistently high and stable stealth value is itself a fingerprinting signal. 0: indistinguishable from a normal browser. 5–15: noticeable modification. 15+: strongly flagged. A stable stealth value of e.g. 20 across all sessions is as identifying as a stable canvas hash.
Trash count Count of API responses that are valid but suspicious — technically correct values that are unexpected in a real browser context. Lower severity than lies. Contributes to reduced trust score. 0–2: normal. Higher values indicate unusual configuration.
Fonts count Number of fonts CreepJS could enumerate. On a typical Linux desktop: 70–100 fonts. Tor Browser enforces a fixed allow-list of ~7 fonts at the engine level. Firefox resistFingerprinting also restricts enumeration. The exact font set is unique to most machines and highly stable — one of the strongest persistent identifiers. 50+: full font set exposed, highly identifying. 10–50: restricted. ≤10: effectively standardised (Tor or strong restriction).
Resistance Which fingerprint resistance mode is active. Seen values: empty (none), Gecko (Firefox resistFingerprinting), Tor Browser. The resistance value is stored in the fingerprint — two browsers with the same resistance string look more alike to a tracker, providing crowd anonymity within that group. The value itself does not mean the browser is well-protected; it means the browser declares a resistance mode. Effectiveness varies widely.

Hash metrics (applies to every hash in sections 2 and 3)

MetricWhat it measuresHow to read it
Volatility Fraction of consecutive session pairs where the hash value changed. Formula: transitions that changed ÷ total consecutive pairs with valid data on both sides. Range 0–1. 0%: hash never changes — a permanent cross-session tracking vector. 100%: different in every consecutive pair — cannot link any two sessions via this signal. Partial values (e.g. 13%) mean occasional rotation — most sessions are still linkable.
Unique ratio (distinct / total) Number of distinct hash values seen ÷ total sessions with valid (non-null) data for that field. Complements volatility: volatility measures how often the hash changes session-to-session; unique ratio measures total variety across all sessions. 1/N: only one value ever — perfectly stable identifier. N/N: every session a different value — fully random, no cross-session linkage. E.g. 4/23 means 4 distinct values across 23 sessions — the hash occasionally rotates but still links most sessions together.
Tracking risk label Summary classification derived from unique ratio: stable identifier (1 unique value = 1/N), partially stable (some rotation, unique ratio 10–84%), fully rotating (≥85% unique). A "stable identifier" signal can be used by any tracker to permanently link all sessions with high confidence, regardless of what other signals do.

Summary hashes — what each one fingerprints

HashWhat it capturesStability on unmodified browserPrimary anti-FP target?
Canvas 2D Pixel data rendered by the 2D canvas API (getImageData after drawing text/shapes with specific styles). Differences arise from GPU model, GPU driver version, OS font rendering, and sub-pixel antialiasing. Two machines with the same GPU and driver version produce identical hashes. Extremely stable — same value across reboots, browser restarts, and browser upgrades on the same machine. Changes only with GPU driver update or hardware change. Yes. CanvasBlocker randomises this per-session. Tor Browser homogenises it to one value shared by all Tor users.
WebGL Pixel data rendered via WebGL shaders, combined with GPU capability strings (RENDERER, VENDOR). Even more hardware-specific than Canvas 2D. The renderer string (e.g. "Intel(R) HD Graphics 520 SKL GT2") directly names the GPU model and driver version. Extremely stable. The renderer string alone is identifying regardless of the pixel hash. Yes. Targeted by CanvasBlocker and Tor Browser.
Audio Floating-point audio samples generated by the OfflineAudioContext API after running a specific oscillator through a compressor node. Output depends on CPU floating-point unit behaviour and OS audio stack. Very stable — identical across sessions on the same hardware. Yes. CanvasBlocker adds noise to audio output. Can also be spoofed via Math interception.
Fonts Hash of the exact set of fonts CreepJS can enumerate using the font access API and canvas text measurements. The installed font set is unique to most machines and reflects software history (installed apps, OS, user customisation). Highly stable — changes only when fonts are installed or removed. Tor Browser enforces a fixed allow-list. Firefox resistFingerprinting restricts enumeration to a smaller set.
Navigator All navigator object properties: userAgent, platform, language, plugins, hardwareConcurrency, maxTouchPoints, deviceMemory, doNotTrack, connection info, and more. Stable — changes with browser version upgrades or OS changes. Tor Browser spoofs hardwareConcurrency, platform. resistFingerprinting modifies userAgent, language, others.
Screen Screen dimensions (width, height, colorDepth, pixelDepth, availWidth, availHeight) and window dimensions. Relatively low entropy — many users share common resolutions like 1920×1080. Stable unless monitor changes. Low individual entropy but combines with other signals. Tor Browser forces 1000×600 window. resistFingerprinting rounds screen values to common sizes.
Timezone IANA timezone string (e.g. Europe/Warsaw) and UTC offset. Reveals approximate geographic location. Medium entropy — narrowed to a timezone but shared by many users. Stable unless system timezone changes. Tor Browser forces UTC. resistFingerprinting may spoof this depending on version.
Worker Properties accessible from inside a Web Worker thread. Should be consistent with the main thread navigator values. Discrepancies between main thread and worker values are themselves a fingerprinting signal (indicates incomplete spoofing). Stable. Inconsistency with Navigator hash indicates a spoofing tool that covers the main thread but not workers. Spoofing tools must intercept both main thread and worker scope. Many fail to cover workers — CreepJS detects this mismatch.
WebRTC Local IP addresses exposed by WebRTC peer connection negotiation. Can reveal the real local network IP even when behind a VPN or proxy. Null/missing if WebRTC is blocked entirely. Stable on the same network. Changes when network changes. Often blocked entirely by privacy browsers. Tor always blocks. A null value here is itself expected and identifying for Tor/privacy users.
Status Browser status flags and feature support indicators — which APIs exist, which permission states are active, which features are enabled/disabled. Stable across sessions on the same browser version and configuration. Low priority target, rarely spoofed.

It does not identify you individually within that group, but it definitively identifies you as a CanvasBlocker user — and the group is far smaller than the general population.

FeaturesWhich browser APIs are present and at what capability level — reflects browser build, enabled feature flags, and extensions that inject new global APIs.Medium — stable per browser build. Extensions adding global APIs are detectable here, potentially identifying specific extensions.

Lie categories — what each API class means when it appears

API ClassWhat it isWhy it appears in lies
FunctionThe built-in Function class, specifically Function.prototype.toString().JS Proxy objects return a different toString() than native functions. This is the primary indicator of proxy-based spoofing — if this appears, every other lie is almost certainly a proxy intercept as well. Its presence means CreepJS has positively identified a spoofing tool using JS Proxies.
CanvasRenderingContext2DThe 2D canvas drawing API — getImageData, fillText, measureText, drawImage, and related methods (9 total).Anti-fingerprinting tools intercept these to return randomised pixel and measurement data. CreepJS detects the intercept by calling the same method through multiple independent call paths and comparing results.
HTMLCanvasElementThe <canvas> DOM element itself — toDataURL, toBlob, captureStream (9 methods).Intercepted alongside CanvasRenderingContext2D by canvas-spoofing tools.
OffscreenCanvasRenderingContext2D / OffscreenCanvasThe off-screen canvas API — runs canvas operations without a visible DOM element, often used in Web Workers.Tools that only intercept the main canvas miss the off-screen variant, producing a detectable inconsistency. Tools that cover both still trigger the Function.toString lie.
AnalyserNode / AudioBuffer / BiquadFilterNodeWeb Audio API nodes for signal analysis, buffered audio data, and audio filtering.Intercepted by audio fingerprint spoofing tools. Same proxy detection mechanism as canvas.
WebGLRenderingContext / WebGL2RenderingContextWebGL 1 and 2 rendering contexts — getParameter, getSupportedExtensions, rendering methods.GPU fingerprint spoofing intercepts these to change the renderer string and pixel output.
NavigatorThe navigator object — 22 properties including userAgent, platform, hardwareConcurrency.Intercepted to spoof platform, core count, or user agent. Partial coverage (e.g. spoofing main thread but not worker) produces detectable inconsistencies.
ScreenThe screen object — 15 properties including width, height, colorDepth, availWidth.Intercepted to hide real screen resolution. Tor does this at engine level; extensions use proxies, making it detectable.
MathJS Math functions — 19 methods including sin, cos, atan2, log, exp.Some tools intercept Math to reduce CPU fingerprinting entropy. Extremely difficult to do cleanly — almost always produces detectable inconsistencies because Math is called in so many places.
Date / DateTimeFormat / RelativeTimeFormatDate object and Intl date/time formatting APIs.Intercepted to hide timezone. If timezone spoofing is inconsistent between Date and Intl.DateTimeFormat, CreepJS detects the mismatch.
TextMetricsFont measurement results returned by canvas.measureText() — width, height, bounding boxes.Part of canvas spoofing — font metrics are randomised alongside pixel output by canvas-targeting tools.
DOMRectReadOnly / DOMRect / SVGRectLayout rectangle objects returned by getBoundingClientRect(), getClientRects(), and SVG equivalents.Intercepted by tools spoofing Client Rects. The SVG variant is often missed, causing a detectable mismatch between HTML and SVG rectangle measurements.
Document / Element / Node / HTMLElement / HTMLIFrameElementCore DOM classes — the base object types for all page elements.Intercepted by tools broadly modifying DOM properties for layout or fingerprint resistance. Often a sign of over-broad proxy application.
MediaDevicesThe API for enumerating cameras, microphones, and other media input devices.Intercepted to hide the real device list. A spoofed device list that is inconsistent with other signals (or returns an implausible configuration) is detectable.
FontFaceThe CSS Font Loading API — programmatic font loading and measurement.Intercepted by font spoofing tools. Inconsistency between FontFace results and canvas font measurements is detectable.
IntersectionObserverEntryLayout intersection data from the Intersection Observer API — reports how much of an element is visible.Intercepted by some tools that broadly proxy DOMRect-related objects.
CSSStyleDeclaration / Range / Permissions / StorageManagerMiscellaneous DOM and browser APIs.Less common intercept targets. Appear when a tool over-broadly proxies entire DOM class prototypes rather than targeting specific methods.
